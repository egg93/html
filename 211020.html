36. UI vs API

UI: User Interface
API: Application Programming Interface

UI => 사용자 조작 장치
웹페이상에 보여지는 버튼의 사용자 => 웹 또는 앱을 이용하려는 사용자

API => 개발자 조작 장치
버튼을 클릭하면 나오는 경고창은 누가 만들었는가 => 웹, 앱 개발자가 만들기도 웹, 앱 개발자가 만들지 않기도
경고창을 띄우기 위해 사용하는 조작 장치, alert() 함수는 웹 브라우저를 개발하는 사람들이 일반 개발자 대신에 만들어 둔 조작 장치. 일반인이 사용 X, 코딩하는 사람이 사용.

순서가 없다면 API도 소용이 없고, API가 없다면 순서도 소용이 없다.
API와 순서는 마치 단어와 문법같은 관계.

37. 수업을 마치며

공부보다는 실습, 실습보다는 자신의 프로젝트를 짜는 것이 중요.
공부에는 때가 없지만 프로젝트를 시작하는 것엔 때가 있다.
지금 이 순간이 본인의 프로젝트를 시작할 때.

시작이 늦어질수록, 공부를 많이 할수록 자신이 짠 코드를 긍정하기 어려워진다. 머릿속이 복잡해진다.

조언
1. 자신의 프로젝트를 시작할 때 모든 개념을 총동원하려고 하지 말 것.
필수불가결한, 최소한의 도구만을 가지고 문제를 해결하려고 시도해볼 것.

2. 최소한의 도구? "Programming"이라는 언어 속에 이미 함유.
순서에 따라 실행되어야 하는 명령들이 실행되도록 하는 것.
객체, 함수, 조건문, 반복문 필요 X.
순서대로 실행된다. 가 중요.

3. 최소한의 지식으로 현실의 문제를 해결하기 위해 노력해보는 것.
그러다 보면 그것만으로는 도저히 해결할 수 없는 순간이 옴.
그 때에 주의깊게 조건문, 반복문, 함수, 객체를 신중하게 도입하면서 각각의 개념들 간에 관계에 익숙해지다 보면 마침내 유창하게 이런 개념들을 활용할 수 있게될 것.

그러다가 또다시 한계에 부딪히게 되면 그때가 실습을 멈추고 공부를 다시 시작하기 좋은 때.

예측되는 한계
- 웹 페이지에 있는 어떠한 태그를 삭제하거나 자식 태그를 추가하고 싶다면,
=> document 객체를 조금 더 자세하게 살펴볼 것. 그 안에 필요한 메서드가 포함되어 있을 것.
=> 2차 해결안: DOM으로 수색해볼 것. (Document Object Model)
(Document 객체는 DOM 객체의 일부)

- 웹 페이지가 아닌 웹 브라우저 자체를 제어해야 한다면,
=> window 객체를 조사해볼 것.
예를 들어, 현재 열려 있는 웹 페이지 주소가 무엇인지 알아내야 하거나, 새 창을 열어야 하거나, 웹 브라우저 화면 크기를 js를 통해 알아내야 한다면,
=> window 객체에 속해 있는 property나 method가 도와줄 수 있을 것.

- 웹 페이지를 리드하지 않고 정보를 변경하고 싶다면,
=> ajax (현대적인 웹, 앱을 만드는데 필수적)

- 웹 페이지가 리로드되어도 현재 상태를 유지하고 싶다면,
=> cookie에 대해 배워볼 것
=> 사용자를 위한 개인화된 서비스 제공 가능

- 인터넷이 끊겨도 동작하는 웹 페이지를 만들고 싶다면,
=> offline web application

- 화상통신 web app을 만들고 싶다면,
=> webRTC

- 사용자의 음성을 인식하고 음성으로 정보를 전달하고 싶다면,
=> speach로 시작하는 api들을 살펴볼 것.

- 3차원 그래픽으로 게임과 같은 것을 만들고 싶다면,
=> webGL

- 가상 현실에 관심이 많다면,
=> webVR
